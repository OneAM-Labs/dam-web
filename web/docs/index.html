Introduction

DAM is a modern developer platform that evolved from a Version Control System (VCS) into a comprehensive Project Continuity Platform (PCP).

Beyond Version Control

Traditional systems ask: "What changed?" DAM asks: "How can this project be preserved, understood, recreated, and continued anywhere?" Version control is just one component of continuity.

Philosophy

DAM approaches project management as critical infrastructure, using terminology inspired by water control systems. A real dam quietly, reliably controls enormous amounts of water. DAM reliably handles your project state.

Instead of patching disjointed tools together, DAM unifies version history, collection rules, environment metadata, and project setup into a single Reservoir.

Core Vocabulary

DAM is not built on top of Git, and its terms aren't a reskin of Git's — they describe DAM's own model of a project's continuity. If a word below reminds you of something from another VCS, that's coincidental; treat this as the primary definition.

Term Definition
Reservoir The totality of a project's continuity: history, configuration, and state. Created the moment you run dam source.
Collection A holding pool of files you've selected to be saved in your next snapshot, governed strictly by purities rules.
Seal An immutable, preserved historical snapshot of everything currently in your collection.
Timeline The full chronological history of seals on your current stream.
Stream An independent, parallel flow of development you can switch between.
Flowinto The act of shifting your active workspace into a different Stream.
Apply Restoring your workspace files to match a specific seal.
Sync Pushing or pulling seal history and file objects between your local reservoir and a linked cloud platform, such as GitHub.
Drain Clearing a section of your reservoir, such as emptying the collection area.
Export Packaging a seal or your entire project into a portable archive for sharing.
Provider A project-type profile (Flutter, Custom, etc.) defining environment checks and setup commands for project exports.

Quick Start

Begin managing project continuity in seconds.

1. Source a Reservoir

Navigate to your project folder and initialize a new continuity environment. This launches an interactive setup wizard for the project name, conflict resolution preference, and an optional dam.toml profile. 

$ dam source
[Success] Reservoir initialized successfully.

You can also skip the prompts entirely:

$ dam source --name "My Project"

2. Define purities

Create a .purities file. DAM operates on strict allowlists by default. Only explicitly declared files or patterns are collected.

# .purities
src/**
README.md
package.json

3. Collect State

Gather your files based on the defined rules.

$ dam collect .
[Success] Collected 14 items matching purities rules.

4. Seal the State

Create an immutable snapshot of your collected environment. The collection area is cleared automatically right after.

$ dam seal "Established core project structure"
[Success] seal_001 created safely.

5. Sync to the Cloud (optional)

Back your reservoir up to a GitHub repository so it's safe off your machine and pullable from anywhere else.

$ dam sync --action push
✓ Successfully pushed seal snapshot to remote repository on GitHub!

.purities & .impurities

DAM introduces two complementary tracking systems to replace error-prone exclusion lists. These are plain-text rule files placed anywhere in your project folder tree, using patterns similar to .gitignore syntax. Lines starting with # are comments.

  • .purities: Acts as a strict allowlist. If a directory contains this file, it enters strict allow mode — only files matching its patterns (within that folder and below) are ever collected.
  • .impurities: Acts as a blocklist, inheriting downwards. Files matching its patterns are excluded, even if otherwise eligible.

Example .impurities file:

# Skip build artifacts and secrets
build/
*.log
.env

If a file matches both a purity and an impurity rule, that's a conflict. DAM detects the collision and resolves it based on your reservoir's settings (dam settings) or a one-time override via dam collect --rule-priority. If no priority is configured, DAM shows you the conflicting files and asks how to proceed, optionally saving your choice as a permanent preference.

Project Profiles (dam.toml)

For full project exports (as opposed to single seal exports), DAM uses a dam.toml configuration file in your project root. This one file is what lets a single command — dam export project — package not just your files, but everything needed to get the project running again somewhere else. It defines:

  • Project info — name and provider type.
  • Setup commands — instructions to run after importing (e.g. installing dependencies).
  • Profiles — named sets of include/exclude file patterns, letting you create different export packages for different audiences (e.g. a full profile for admins, and a contributor profile that excludes sensitive files like .env).

Example dam.toml:

[project]
name = "my-app"
provider = "flutter"

[setup]
commands = [
    "flutter pub get",
    "flutter pub run build_runner build"
]

[profiles.full]
description = "Everything, for internal team members"
include = ["**/*"]
exclude = [".env", "*.log"]

[profiles.contributor]
description = "Safe to hand to an external contributor"
include = ["lib/**", "pubspec.yaml", "README.md"]
exclude = [".env", "**/secrets/**"]

Providers are project-type templates that come with sensible defaults, driving two things automatically from that single config:

  • Environment checks — before unpacking a .dam archive, DAM asks the declared provider whether your machine looks ready for this kind of project (e.g. is a required toolchain installed), and warns you if it isn't.
  • Setup commands — the commands listed in [setup] travel with the archive. On import, DAM surfaces them for you to run so your dependencies get installed; exactly how they're surfaced depends on the provider (see below).
Provider Best for On import
flutter Flutter/Dart mobile projects. Checks that Flutter is installed, then prints the declared setup commands as ready-to-run suggestions.
custom Any other type of project, or one with no standard toolchain to verify. No environment check is performed. Setup commands are written to a setup_untrusted.sh script in the extracted project instead of being printed, so you can review arbitrary commands before running anything.
Note

DAM never executes setup commands automatically, regardless of provider — either it prints them for you to run yourself, or it writes them to a script for review first. This keeps importing someone else's project from silently running arbitrary code on your machine.

You can generate a dam.toml automatically during dam source, or by upgrading with dam source --upgrade.

GitHub Integration

DAM can target a GitHub repository as a cloud remote for your reservoir. Once configured, dam sync translates your local seal history, workspace objects, and stream metadata into Git database structures and pushes them to GitHub. It can also pull remote changes down to any machine—turning GitHub into an off-site backup, collaboration hub, and sharing point.

Target repository

DAM reads the github_repo reservoir setting from your local .dam/config.toml (using the format owner/repo_name). If this setting is not configured, DAM will interactively prompt you for the target repository during your first sync and save your input automatically. 

dam settings github_repo "your-username/your-project"

Authentication

DAM resolves your GitHub credentials by checking the following, in order:

  1. Structured Credentials: It attempts to load structured JSON credentials from .dam/credentials.json.
  2. Legacy Fallback: If no JSON file is found, it checks for a legacy .dam/credentials file containing a raw token (treated as a Classic Personal Access Token).
  3. Interactive Prompt: If no stored credentials exist, DAM presents an interactive menu allowing you to choose your authentication mechanism:
    • Fine-Grained Personal Access Token (Recommended)
    • Classic Personal Access Token
    • SSH Key Path (Note: Stored for future native SSH transport; the current REST API provider requires a token).

Once entered, your credentials are securely structured and saved to .dam/credentials.json so you won't be prompted again.

Note

.dam/credentials.json stores your authentication details in plain text on disk. Keep it out of anything you export or share—project exports never include your .dam directory, only your working files, so this is safe by default as long as you do not manually collect or seal your .dam directory.

Stream-Aware Branch Mapping

DAM sync is natively stream-aware. When you synchronize, local streams are mapped directly to corresponding Git branches on your GitHub repository (refs/heads/{stream}):

  • Targeted or Bulk Syncing: You can pass a specific stream name to sync just that stream, or run a general sync to iterate through all streams located in .dam/streams.
  • Main Stream Protection: Because main is intended to remain your stable development stream, attempting to synchronize main will trigger an interactive warning prompt unless you bypass it using the `--force` flag.
  • History Stitching: When pushing a brand-new stream to GitHub for the first time, DAM dynamically checks the remote SHA of the stream it was created from (its base target, defaulting to main). It attaches this parent SHA to the new commit so that GitHub Pull Request diffs connect cleanly.
  • Empty Repo Bootstrapping: If you push to a completely empty GitHub repository, DAM automatically detects the conflict and bootstraps the initial Git database structure with a .dam_keep file before pushing your stream tree.

Interactive Diffs & Conflict Resolution

You can execute explicit push or pull actions, or let DAM run an interactive state diff by comparing your local seal lineage against the remote branch tree:

  • Up to Date: If no differences are found between local and remote seals, DAM skips the stream.
  • Ahead or Behind: If your local stream is ahead (new local seals) or behind (missing remote cloud seals), DAM displays the exact count and prompts you to confirm pushing or pulling.
  • Conflict Detection: If a stream has diverged (both ahead and behind), DAM alerts you to the conflict and provides an interactive menu to either [1] Pull remote changes or [2] Force Push local state.

When pulling changes, DAM selectively downloads only the missing compressed objects into .dam/objects/, writes the missing seal records to .dam/seals/, and automatically updates your local stream metadata to point to the newest head seal.

Command Reference

Every DAM command, what it does, and the options it accepts.

dam source

Initializes a brand-new reservoir in the current directory, or upgrades an existing one.

dam source [--name <NAME>] [--upgrade]
  • --name, -n — Skip the name prompt and use this name directly.
  • --upgrade — Migrate an existing reservoir to the latest DAM format. Safe to run any time; it won't touch your history.
Note

If a reservoir already exists in a parent folder, DAM warns you before creating a separate nested one, so you don't end up with confusing overlapping projects.

dam collect

Scans a folder (or single file) and adds matching items to your collection, ready to be sealed.

dam collect <PATH> [--override-purities [FILE]] [--override-impurities [FILE]] [--rule-priority <purities|impurities>]
dam collect .                          # Scan everything in the current folder
dam collect src/                       # Scan only the "src" folder
dam collect . --override-purities      # Ignore all .purities allowlists for this run
dam collect . --rule-priority purities # Force purities to win any conflicts
  • --override-purities — Without a value, ignores all .purities rules entirely. With a file path, uses only the rules in that file.
  • --override-impurities — Same idea, but for .impurities blocklists.
  • --rule-priority — Decides who wins when a file matches both a purity and an impurity rule.

dam creds

Securely stores, lists, and deletes credentials using your operating system's keychain or an encrypted local vault.

dam creds create [--alias <NAME>] [--vault]
dam creds list [--vault]
dam creds delete <ALIAS> [--vault]
dam creds create
# Interactive setup for a new credential

dam creds create --alias github_personal
# Save a credential with a custom alias

dam creds create --alias github_work --vault
# Store the credential in the encrypted local vault instead of the OS keychain

dam creds list
# Show all registered credential aliases

dam creds delete github_personal
# Remove a credential by alias

dam creds delete github_work --vault
# Remove a credential from the encrypted local vault
  • create — Interactively creates a new credential. You'll be prompted to choose a credential type (GitHub fine-grained token, classic token, SSH key, or generic secret) and provide the required value.
  • list — Displays all registered credential aliases without revealing the stored secrets.
  • delete — Permanently removes the credential associated with the specified alias.
  • --alias — Specifies the name used to reference the credential later. If omitted, you'll be prompted to enter one.
  • --vault — Stores or manages credentials in DAM's encrypted local vault instead of your operating system's secure credential store.

dam inspect

Your dashboard for seeing the current state of the reservoir.

dam inspect [--rules] [<TARGET>]
  • By default — shows the currently active stream and all files sitting in the collection area.
  • With --rules — a complete map of every .purities and .impurities file found in your project, plus a summary count of each.
  • With a target ending in .dam — inspects a packaged project archive and prints its metadata without extracting it.

dam drain

Empties a specific part of your reservoir.

dam drain <TARGET>

collection — Clears the collection area back to empty. This does not touch your actual workspace files or any sealed history.

dam seal

Creates a permanent snapshot from everything currently in the collection area — the equivalent of a commit.

dam seal "<MESSAGE>"
dam seal --list <N>
dam seal "Added user login flow"
dam seal --list 5     # Show the 5 most recent seals
Notes

A seal records the exact content of every staged file, your message, and a timestamp. After sealing, the collection area automatically empties. Each seal gets an incrementing ID like seal_001, seal_002, and so on.

dam timeline

Prints the chronological history of seals on your current stream. You can also render a visual ASCII Directed Acyclic Graph (DAG) to view branches, merge splits, and stream relationships.

dam timeline [--graph]
dam timeline          # Print the full chronological text log
dam timeline --graph  # Render the topological ASCII DAG visualization
  • --graph, -g — Display a graphical ASCII representation of your reservoir continuity, highlighting the active stream head and merge connections.

dam stream

Manage parallel development streams (the DAM equivalent of branches). DAM streams support rich continuity metadata including descriptions, ownership, priorities, targets, goals, and notes.

dam stream [SUBCOMMAND]
dam stream                                         # List all streams and their priority/status
dam stream create experiment-ui                    # Create a stream inheriting current history
dam stream create feature-login --from main        # Branch explicitly from the 'main' stream
dam stream inspect experiment-ui                   # View detailed metadata, goals, and latest seal
dam stream goal experiment-ui "Refactor navbar"  # Add a specific goal to the stream
dam stream delete experiment-ui                    # Delete a stream (prompts if seals are attached)

Subcommands:

  • create <NAME> [--from <STREAM>] — Create a new stream. Automatically inherits the latest seal from the parent stream so historical continuity is preserved. Prompts interactively for metadata if flags are omitted.
  • inspect [<NAME>] — Display stream dashboard including description, owner, priority, target, status, latest seal, and active goals. Defaults to the active stream if omitted.
  • goal <NAME> [TEXT] [--clear] — Add a numbered goal to the stream, or clear existing goals.
  • notes / description / target — Update or clear rich metadata attributes on the specified stream.
  • delete <NAME> — Remove a stream. If the stream has associated historical seals, DAM will warn you and require interactive confirmation before removing them.

dam flowinto

Safely diverts your active workspace flow to a different development stream — the DAM equivalent of checking out a branch.

dam flowinto <NAME>
dam flowinto experiment-ui

If you have unsealed, staged files in your workspace when switching streams, DAM detects the unsaved continuity and will prompt you to automatically generate a temporary Continuity Snapshot (Seal) before flowing into the new stream.

dam merge ⚠️ Experimental, not suggested for production workflow yet

Merge another stream's seals and file structures into your currently active stream using a smart 3-way conflict resolution engine.

dam merge <SOURCE> [--apply]
dam merge experiment-ui          # Calculate merge plan and flag potential conflicts
dam merge --apply                # Apply the calculated merge and generate a multi-parent seal
  • <SOURCE> — The name of the stream you want to merge into your currently active workspace.
  • --apply, -a — Execute the previously calculated merge plan, restore workspace files from the object pool, and commit a new multi-parent seal to history.
Smart 3-Way Merging

When you run dam merge <SOURCE>, DAM climbs backward through your DAG lineage to identify the closest common ancestor (Base Seal). It then evaluates all file paths across the Base, Target, and Source seals:

  • Clean Updates: Changes made purely in the source stream are automatically accepted.
  • Conflict Handling: If a file was modified differently in both streams, DAM marks it as a structural conflict in the merge plan and defaults your active workspace file to your target stream's version.
  • Multi-Parent Sealing: Applying the merge creates a permanent seal linking both the Target Head and Source Head as parents, preserving true continuity.

dam apply

Restores your workspace files to exactly match a previously sealed snapshot.

dam apply <SEAL_ID> [--preview]
dam apply seal_001            # Restore files to match seal_001
dam apply seal_001 --preview  # See what would change, without touching anything
Safety features

With --preview, DAM only prints a description and affected files — nothing on disk is modified. If you have unsealed files in your collection, DAM warns you before restoring and offers to automatically create a safety seal first.

dam sync

Pushes or pulls your seal history, stream metadata, and object blobs to and from a linked cloud platform. GitHub is the default provider.

dam sync [<STREAM>] [--action <ACTION>] [--platform <PLATFORM>] [--force]
dam sync                                  # Interactively diff and sync all eligible streams
dam sync experiment-ui                    # Run interactive diff exclusively for 'experiment-ui'
dam sync --action push                    # Push all stream branches to the cloud
dam sync experiment-ui --action pull      # Pull remote updates for a specific stream
dam sync main --force                     # Bypass the protection warning when syncing 'main'
  • <STREAM> — Optional target stream name. If omitted, DAM scans .dam/streams and synchronizes all eligible streams interactively.
  • --action, -a — Explicitly set to push or pull. If omitted, triggers interactive diff mode.
  • --platform, -p — The cloud provider to target. Defaults to github.
  • --force, -f — Bypass interactive confirmation prompts, specifically when synchronizing the protected main stream.
Interactive Diff & Stream Protection

In interactive mode, DAM compares your local seal lineage against remote branch heads and displays how many seals you are ahead or behind by. If both exist, it alerts you to the diverged branch and lets you choose between pulling remote changes or force-pushing your local state.

Because main is intended to remain a stable development timeline, attempting to sync it will trigger an interactive warning prompt unless you pass the --force flag.

See GitHub Integration for how repository targeting and authentication are configured.

dam export

Packages your project for sharing. There are two distinct modes.

Export a Seal

Share one specific historical snapshot.

dam export seal <SEAL_ID> [--zip]
dam export seal seal_001        # Compact, compressed bundle (.seal) for re-importing into DAM
dam export seal seal_001 --zip  # Plain .zip of the raw files, for general use

Export a Project

Package your entire current project — not just one snapshot — into a portable .dam archive that includes setup instructions and metadata, ready to hand off to someone else. 

dam export project <PROJECT_NAME> [--profile <PROFILE>]
dam export project my-app --profile contributor

This uses the include/exclude rules and setup commands defined in your project's dam.toml profile (see Project Profiles). If your reservoir is configured to require a password for project exports, you'll be prompted to set one, and the archive will be encrypted.

dam import

Imports a previously exported archive back into a reservoir.

dam import <FILE>
  • .seal — A compressed snapshot bundle exported with dam export seal.
  • .zip — A raw, uncompressed file package.
  • .dam — A full portable project archive exported with dam export project. Importing one of these shows the project's metadata, checks whether your machine has the right environment for its provider, prompts for a password if encrypted, and extracts the files into a new folder (or merges into the current one).
dam import teammates-work.seal
dam import shared-app.dam

If the imported project includes custom setup commands, DAM writes them to a reviewable script rather than running them automatically, so you can check what they do first.

dam settings

View or change your reservoir's configuration.

dam settings                            # Print all current settings
dam settings --interactive              # Open an interactive settings dashboard
dam settings <KEY>                      # Read the current value of one setting
dam settings <KEY> <VALUE>              # Update one setting directly
Key Description
name Your reservoir's project name.
purities_overrides_impurities If true, purity (allow) rules automatically win conflicts.
impurities_overrides_purities If true, impurity (block) rules automatically win conflicts.
suppress_nested_warning If true, hides the warning about nested reservoirs.
enforce_password_on_project_import If true, project exports always require a password.
github_repo Target GitHub repository for dam sync, in owner/repo format. See GitHub Integration.
github_token_path Optional path to a file containing your GitHub access token, used instead of the default .dam/credentials location. 
dam settings name "New Project Name"
dam settings purities_overrides_impurities true

dam destroy

Permanently removes — or safely archives — your entire reservoir. Use with care.

dam destroy            # Interactively delete the reservoir
dam destroy --archive  # Rename the reservoir to a backup instead of deleting it
Safety measures

Destruction always shows exactly what will be lost before doing anything. You must type your exact project name (or the word DESTROY) to confirm, followed by a final "are you sure" prompt. The --archive option renames your reservoir to a backup folder instead of deleting it outright.

dam update

Checks for available CLI updates and manages your DAM version manually. Maintains intelligent version synchronization with proactive cache updates and urgency-based prompts.

dam update
dam update --check-only

Running dam update connects to the update server and proactively synchronizes DAM's internal cache, ensuring the background checker is always aware of your manual update attempts. The urgency level of any available release (Soft, Medium, or Urgent) determines whether an interactive prompt appears — Soft updates are silent, while Urgent updates demand immediate attention.

Options

Flag Description
--check-only Check for updates without prompting or applying. Displays available version information and exits.
--force Force an update check even if a check was performed recently. Useful for testing update detection.
--verbose Display detailed update information including cache state, urgency reasoning, and server response details.
--no-interactive Skip all prompts. Applies the update automatically if urgency is Urgent; otherwise exits silently.

Examples

dam update                        # Standard interactive check
dam update --check-only           # Check without prompting
dam update --force                # Force check even if recently checked
dam update --force --verbose      # Force check with detailed information

Background Behavior

DAM automatically checks for updates in the background every 24 hours. Manual dam update attempts reset this cycle, ensuring timely notifications while respecting your preferences. The background checker tracks:

  • Last Check Time: Prevents excessive server requests by maintaining a 24-hour check interval.
  • Urgency State: Remembers the urgency of available updates to determine future prompt behavior.
  • Manual Override Awareness: Acknowledges when you run dam update manually, adjusting prompt scheduling accordingly.
Update Urgency Levels

Soft: Non-critical updates, bug fixes, or minor improvements. No interactive prompt. Medium: Important features or security patches. Interactive prompt shown. Urgent: Critical security fixes or breaking changes. Immediate interactive prompt with emphasis.

FAQ

What's the difference between exporting a seal and exporting a project?

Exporting a seal shares one specific snapshot of your history — useful for sending a single version of a file set. Exporting a project packages your entire current setup, including setup instructions and provider info, meant for someone to pick up and start working with right away.

What happens to my collection when I seal?

It's automatically cleared, so you're ready to start collecting your next round of changes.

Can I undo a destroy?

No — dam destroy is permanent. Use dam destroy --archive instead if you want a safety net; it simply renames your reservoir to a backup folder rather than deleting it.

What if two rule files disagree on a file?

That's a conflict. By default DAM asks you what to do, but you can set a permanent preference with dam settings, or override it per-run with dam collect --rule-priority.

What happened to dam fetch?

It's been replaced by dam sync. Fetch pulled history from another local reservoir path; sync does the equivalent job against a configured cloud platform (GitHub today), and also handles pushing your own history up, which fetch never did.

What's the difference between dam sync and dam export project?

dam sync keeps your full seal history in continuous two-way sync with a cloud platform — it's for backup and picking up work elsewhere. dam export project creates a one-time, self-contained snapshot for handing off to someone who isn't going to use DAM's sync at all.

Is my data encrypted?

  • Credentials are always encrypted (OS Keychain or AES-256-GCM vault).
  • Project exports support optional password encryption.
  • Seal exports and local reservoirs are not encrypted by default.

How do I manage credentials?

Use the dam creds command suite:

  • dam creds create — Store a new credential.
  • dam creds list — Display all registered aliases.
  • dam creds delete — Remove a credential.

What happens to old credential files when I upgrade?

Running dam source --upgrade automatically migrates legacy credentials into the new system and permanently removes the old plaintext files.

How does 3-way merging work?

dam merge finds the closest common ancestor between your current stream and the source stream, then intelligently combines changes. If both sides modified the same file differently, you'll be guided through resolving the conflict.

What's the difference between dam sync and dam export?

dam sync pushes your entire seal timeline and stream metadata to GitHub for backup and sharing. dam export packages a single seal or your entire project into a portable archive for one-time distribution.

What's New

New: Pull Request Workflows

Headline feature: Inspect source PRs in Isolated Local Streams

DAM now makes it easy to review and test open source pull requests locally with the new dam pr command. Browse open PRs from your configured repository and check out contributor changes into a brand-new isolated stream without touching your existing streams.

New: dam pr Command

Use dam pr to interactively browse open pull requests, or use the available subcommands to list PRs and check out a specific pull request directly. 

dam pr                       # Interactively list PRs and choose one to check out
dam pr list                  # List open pull requests
dam pr checkout 19           # Check out PR #19 into stream 'pr-19'

Isolated PR Streams

Checking out a pull request never modifies your existing streams. DAM downloads the PR's head commit into a brand-new stream, giving you a safe environment to inspect, build, and test contributor changes locally.

  • Browse Open Pull Requests: Fetches open pull requests from your configured source and presents them for interactive selection.
  • Dedicated PR Streams: Every checked-out pull request is placed into a new isolated stream, keeping your existing streams and work untouched.
  • Direct Checkout: Check out a specific pull request by number with dam pr checkout <number>. The PR is automatically placed into a dedicated stream such as pr-19.

Safe Handling of Uncommitted Work

DAM protects your local work before switching into a pull request stream. If your active stream contains uncommitted collected files, you'll be offered a safety seal before the PR checkout proceeds. 

# If your active stream has uncommitted collected files:

⚠ Warning: Your active stream contains uncommitted collected files.

A safety seal is required before checking out this pull request.
This ensures your local work is preserved.

Continue with safety seal? [Y/N]
  • Never Lose Local Work: PR checkout cannot silently discard uncommitted collected files from your active stream.
  • Safety Seal Before Checkout: When local work is detected, DAM gives you the opportunity to seal it before switching streams.
  • Non-Destructive Workflow: Your current stream remains intact while you inspect and test the contributor's changes.

Review PRs Without Disrupting Your Workflow

With dam pr, reviewing a contributor's work becomes a straightforward local workflow: find an open PR, check it out into its own stream, inspect or build the changes, and return to your existing work without mixing the two environments.

Warning Suppression Options

  • Permanent Suppression [Y]: Saves your preference to ~/.dam/config/suppressed_warnings.json, ensuring the warning never appears again for this user.
  • Keep Showing [N]: Continues displaying the warning on future DAM operations until you choose to suppress it.
  • Export PATH [E]: DAM automatically detects your shell (bash, zsh, fish, etc.) and outputs the exact line to add to your shell profile, then optionally adds it for you.

Re-enabling Suppressed Warnings

If you need to re-enable a suppressed warning, you can do so through DAM settings:

dam settings --enable-warning path_not_in_env

Or by manually editing ~/.dam/config/suppressed_warnings.json.

v0.5.0

Headline features: Secure Credential Management

v0.5.0 introduces a centralized, secure credential storage system and automatic migration from legacy credential formats.

New: Centralized Credential Manager

DAM now provides a unified system for storing GitHub tokens, SSH keys, and generic secrets securely. Choose between OS Keychain (default) or an encrypted local vault.

dam creds create --alias github_token  # Store securely
dam creds list                          # View all aliases
dam creds delete github_token           # Remove a credential

New: Credential Types Support

When creating credentials, you can choose from four types:

  • Fine-Grained Tokens (GitHub) — Recommended for security; scoped to specific repositories.
  • Classic Tokens (GitHub) — Legacy personal access tokens with broad permissions.
  • SSH Keys — Path to a private SSH key file.
  • Generic Secrets — Any custom API key or secret.

New: Encrypted Local Vault

Use --vault to store credentials in an encrypted local vault instead of OS Keychain. Credentials are encrypted with AES-256-GCM and stored in .dam/vault.bin.

dam creds create --alias my_token --vault  # Encrypt locally
# Password prompted (or via DAM_VAULT_PASSWORD env variable)

New: Automatic Migration Pipeline

Upgrading from v0.4.x automatically migrates legacy credential files:

dam source --upgrade

This migrates .dam/credentials and .dam/credentials.json into the new secure system and permanently removes the old plaintext files.

New: PBKDF2 Key Derivation

Vault passwords are derived using PBKDF2 with SHA-256 (100,000 iterations) for strong key derivation and protection against brute-force attacks.

Enhanced dam sync

GitHub synchronization now utilizes the new credential system automatically. No more manual token management — just create a credential and let DAM handle the rest.

Breaking change: Legacy credential formats removed

The legacy .dam/credentials.json format (from v0.4.x) is no longer supported. Run dam source --upgrade to migrate your credentials to the new secure system.

v0.4.1

Headline feature: Stream-Aware Cloud Sync & Smart 3-Way Merging

v0.4.1 elevates DAM from a local continuity tool into a collaborative, multi-stream ecosystem. With native stream-to-branch mapping for GitHub, visual ASCII DAG graphs, and an intelligent 3-way merge engine, managing parallel development lines is now effortless.

New: dam merge ⚠️ Experimental, not suggested for production workflow yet

You can now merge parallel development streams into your active workspace using a smart 3-way conflict resolution engine. By climbing backward through your history to find the closest common ancestor (Base Seal), DAM evaluates changes across the Base, Target, and Source streams.

  • Smart Resolution: Clean modifications are accepted automatically, while structural conflicts are flagged in a generated merge plan without destroying your active files.
  • Multi-Parent Sealing: Applying a merge (dam merge --apply) creates an immutable seal linking both the Target Head and Source Head as parents, preserving true historical continuity.
dam merge experiment-ui          # Calculate merge plan and flag potential conflicts
dam merge --apply                # Apply the calculated merge and commit a multi-parent seal

See dam merge in the Command Reference for full usage details.

New: Stream-Aware Cloud Sync & Main Protection

dam sync has been completely rebuilt to understand streams. When synchronizing with GitHub, local streams map 1:1 to remote Git branches (refs/heads/{stream}).

  • Interactive Diffs & Conflict Resolution: Running dam sync without arguments compares your local seal lineage against remote branch heads. If a branch has diverged, DAM presents an interactive menu to either pull remote updates or force-push local state.
  • Main Stream Protection: Because main is meant to remain your stable timeline, attempting to sync it triggers an interactive warning prompt unless bypassed with the --force flag.
  • History Stitching & Bootstrapping: Pushing a new stream dynamically attaches the parent SHA of the branch it was created from for clean Pull Request diffs. If pushing to a completely empty repository, DAM automatically bootstraps the initial Git database structure with a .dam_keep file.

New: ASCII DAG Visualizer

You can now visualize your reservoir's topological history directly in the terminal. Passing the `--graph` flag to the timeline command renders an ASCII Directed Acyclic Graph (DAG) displaying branches, merge splits, and active stream heads.

dam timeline --graph

New: Rich Stream Metadata & Goals

Streams are no longer just simple pointers. They now support rich continuity metadata, allowing you to track descriptions, owners, priorities, target branches, notes, and numbered goals directly inside your reservoir.

dam stream create feature-login --from main        # Branch explicitly from 'main'
dam stream goal feature-login "Refactor auth API"  # Add a numbered goal
dam stream inspect feature-login                   # View the rich metadata dashboard

New: Auto-Snapshotting in dam flowinto

Switching streams is now safer than ever. If you attempt to run dam flowinto <STREAM> while having unsealed, staged continuity in your active workspace, DAM detects the unsaved progress and interactively prompts you to generate a temporary Continuity Snapshot before moving your flow.

Breaking change: Modernized Authentication & Removed Settings

The legacy github_token_path reservoir setting and plain environment variable lookups have been removed in favor of a secure, structured credential store. DAM now reads from .dam/credentials.json (with a fallback to .dam/credentials). If no credentials exist, an interactive wizard prompts you to choose between Fine-Grained Personal Access Tokens, Classic Tokens, or SSH key paths.

Breaking change: dam fetch has been removed

dam fetch is no longer part of DAM. Its role—bringing in history and file objects from remote sources—is handled entirely by dam sync pull, which operates natively against your configured cloud platform.

v0.3.0

Headline feature: Cloud Sync with GitHub

v0.4.1 introduces dam sync — the ability to push and pull your seal history to and from a linked cloud platform. GitHub is the first supported platform, with more planned.

New: dam sync

DAM reservoirs can now stay backed up and shared through a real cloud remote. Running dam sync with no arguments compares your local timeline against the remote and walks you through resolving any difference — pushing, pulling, or resolving a conflict if both sides have moved on.

dam sync                        # Interactive: compares local vs. remote, then prompts
dam sync --action push                   # Push local seals straight to the cloud
dam sync --action pull                   # Pull remote seals straight into your reservoir
dam sync --action push --platform github # Explicit platform selection (github is currently the default)

See dam sync in the Command Reference and GitHub Integration for the full breakdown of how authentication and repository targeting work.

New: GitHub as a first cloud platform

DAM can now target a GitHub repository as a remote for your reservoir. Your seal history and the underlying file objects they depend on are translated into commits on the repository, and can be pulled back down on any other machine. See GitHub Integration for setup and authentication details.

New: github_repo and github_token_path settings

Two new reservoir settings support the sync workflow: github_repo stores your target repository (owner/repo), and github_token_path optionally points DAM at a custom file containing your access token. Both are configurable via dam settings, including the interactive dashboard.

Breaking change: dam fetch has been removed

dam fetch is no longer part of DAM. Its role — bringing in history and file data from elsewhere — is now covered by dam sync pull, which does the same job against a configured cloud platform instead of a local path.

Documentation change: DAM now has its own vocabulary

The documentation no longer maps DAM's terms against Git's. DAM is not a Git wrapper, and its terminology is defined on its own terms — see Core Vocabulary.

v0.2.0

Headline feature: Project Export

v0.2.0 introduced dam export project and dam import support for .dam archives — the ability to package your entire project, not just a single snapshot, into one portable file someone else can pick up and run.

Project Export, in full

Previously, DAM could only export individual seals. A seal export is great for sharing one snapshot of files, but it doesn't capture how to actually get a project running again. Project Export closed that gap.

Running dam export project <NAME> [--profile <PROFILE>] bundles:

  • Your current working files, filtered through the include/exclude patterns defined in the selected dam.toml profile (e.g. a contributor profile that strips out .env and other sensitive files).
  • Project metadata — name, author, export timestamp, and declared provider.
  • Any setup commands needed to get the project running again after import.
  • Optional password-based encryption, automatically required if your reservoir has enforce_password_on_project_import enabled. 
dam export project my-app --profile contributor

New: dam.toml project profiles

Project Export is powered by a dam.toml file in your project root, which can be scaffolded automatically during dam source or generated later with dam source --upgrade. It defines named profiles (different include/exclude sets for different audiences) and a provider — which tells DAM what environment checks and setup commands apply on import. See Project Profiles for the full breakdown.

New: smarter dam import

dam import recognizes three formats — .seal, .zip, and the .dam project archive. Importing a .dam file shows you the project's metadata up front, checks whether your machine has the right environment for its declared provider, prompts for a password if the archive is encrypted, and surfaces any custom setup commands for you to review rather than running them blindly.

New: dam inspect on archives

dam inspect can target a .dam file directly, printing its metadata (project name, author, timestamp, provider, profile, setup commands) without extracting anything — useful for vetting an archive before you trust it.

Also in this release

  • dam collect --rule-priority for one-off conflict resolution between .purities and .impurities without changing your saved settings.
  • dam settings --interactive, a numbered-menu dashboard for toggling every setting without remembering exact key names.
  • dam destroy --archive, a safer alternative to permanent deletion that renames your reservoir to a backup folder instead.
